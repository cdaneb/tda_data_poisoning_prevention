**Drift found vs. prior documentation:**

- **`CLAUDE.md` says the TDA pipeline is "1500→72 features (Algorithm 1 from paper)."** The actual code produces **60 features**, not 72. Verified directly from `tda_pipeline.py`: 5 filtrations (2 `HeightFiltration` + 3 `RadialFiltration`) × 6 extractors per filtration (1 `PersistenceEntropy` + 5 `Amplitude` metrics) × 2 homology dimensions (H0, H1, the `CubicalPersistence` default) = 5 × 6 × 2 = **60**. `tda_pipeline.py`'s own `__main__` block explicitly checks for 72 and prints a warning if the count differs — so the code itself acknowledges the mismatch, but `CLAUDE.md` states 72 as fact. This is a live inaccuracy in the file Claude Code loads into context every session.
- **`CLAUDE.md` attributes DPDC/sanitized-purity/poisoned-pool-precision computation entirely to `clustering.py`.** In reality `clustering.py::classify_clusters()` only computes per-cluster `dpdc`; sanitized purity and poisoned-pool precision are computed downstream in `run_iterative.py`/`run_multi_seed.py`, not in `clustering.py`. Minor imprecision, not a functional issue.
- `cc_summary.md` (a prior Claude-generated summary, not authoritative project docs) already correctly documents the 60-feature count and the dependency graph above — it does **not** exhibit the same drift as `CLAUDE.md`. It was generated by a Claude session per `cc_summary_generator.md` and reads as accurate against current source, but see §7 on whether it should stay in root.

## 4. Duplicate logic

- **Minor duplication: TDA "diagram step" pipeline construction.** `tda_pipeline.py::build_tda_pipeline()` builds 5 parallel pipelines of the pattern `Binarizer → Filtration → CubicalPersistence → Scaler`. `iterative_filter.py::compute_persistence_diagrams()` independently constructs **the same 4-step pattern** (`Binarizer(threshold=0.4) → HeightFiltration(direction=[0,1]) → CubicalPersistence() → Scaler()`) inline, rather than reusing a shared helper. It's not a full reimplementation of the whole pipeline (it deliberately uses only 1 of the 5 filtrations, for a different purpose — Wasserstein convergence tracking, not feature extraction) but the boilerplate for a single diagram-step is duplicated. **Canonical version:** `tda_pipeline.py`'s `diagram_steps` construction is the fuller/more general version; `iterative_filter.py`'s copy is a special case of it.
- **Minor duplication: ad hoc numpy→JSON serialization helpers.** `run_iterative.py` defines its own `convert_for_json()` helper (handles `np.integer`, `np.floating`, `np.ndarray`, `np.bool_` recursively via `json.dump(..., default=convert_for_json)`). `run_multi_seed.py` instead uses an inline `lambda x: float(x) if isinstance(x, (np.floating, np.integer)) else x` for one of its two `json.dump` calls, and no custom serializer at all for the other (the aggregated-results dump, which only contains plain Python floats already). These are two different, independently-written solutions to the same small problem. Neither is imported by the other. No shared module currently exists for this.
- Also: `iterative_filter.py` imports `build_tda_pipeline` and `reshape_for_tda` from `tda_pipeline.py` but never actually calls either — only `extract_tda_features` is used. Not duplication, but worth noting as an unused import.
- No other duplication found. `classify_clusters`, `run_all_clustering`, `poison_dataset`, and `extract_tda_features` each have exactly one implementation, correctly imported by all consumers.

## 5. Documentation/code mismatches

1. **`clustering.py` module docstring mentions a "Blue" cluster category** ("contains complete clean AND complete poisoned subsets") that is **never assigned** in `classify_clusters()` — confirmed by reading the function: it only ever assigns `"Noise"`, `"Green"`, `"Red"`, `"Pink"`, or `"Yellow"`. The `summary["colors"]` dict in the same function also only tallies `["Green", "Red", "Yellow", "Pink", "Noise"]` — "Blue" isn't even a key checked for. This is a pure docstring/code drift with no functional impact.
2. **`CLAUDE.md` claims 72 output features from the TDA pipeline; the code produces 60.** See §1 — this is the more consequential mismatch since `CLAUDE.md` is auto-loaded into every session's context, unlike `clustering.py`'s docstring.
3. `CLAUDE.md`'s architecture description of `iterative_filter.py` says it "removes Red clusters into a 'poisoned pool'" — technically accurate but incomplete, since the same iteration also removes Green clusters into a sanitized pool (which the rest of `CLAUDE.md`'s own data-flow diagram correctly implies via "→ results/*.json"). Not a factual error, just an incomplete summary.
